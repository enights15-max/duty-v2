# Plan de Trabajo Release 1.0

Periodo: miercoles 8 de abril de 2026 a sabado 11 de abril de 2026  
Zona horaria: America/Santo_Domingo  
Objetivo: publicar `DUTY 1.0` esta semana con scope congelado y riesgo controlado.

## Regla principal de esta semana

Durante este plan:

1. no abrimos features nuevas;
2. no metemos `Ticket Rewards`, `Sponsors`, `Promoter`, `Memberships` ni `POS` en el release;
3. cualquier trabajo fuera del scope `1.0` se mueve a `Phase 2`;
4. toda decision diaria termina en:
   - `GO`
   - `GO con follow-up`
   - `NO-GO`

## Scope congelado de 1.0

### Dentro de 1.0

1. auth / onboarding base
2. home / search / explore
3. event details
4. checkout principal
5. my tickets / ticket details
6. wallet base
7. scanner base
8. reservations
9. professional create/edit/manage event
10. admin settlement / refunds / collaborator tooling ya estable

### Solo si pasa smoke real

1. wallet topup real
2. mixed checkout real
3. marketplace
4. loyalty redemptions

### Fuera de 1.0

1. Ticket Rewards end-to-end
2. claim scanner de rewards para staff
3. Sponsors
4. Promoter / co-organizer expandido
5. Structured Event Details
6. Community / Memberships
7. POS independiente

## Criterio de salida de la semana

Se publica `1.0` el sabado 11 de abril de 2026 si:

1. el dry run real de staging sale verde;
2. Stripe real queda confirmado;
3. no hay `P0`;
4. no queda `P1` bloqueante de tienda o pago;
5. QA manual minima pasa;
6. build release Android e iOS quedan generados o al menos listos para submission.

Si cualquiera de esos falla, la salida correcta es:

1. no publicar;
2. congelar cambios;
3. cerrar blockers y mover submission al siguiente bloque.

## Plan de contingencia

Si el miercoles o jueves aparece un bloqueo serio en:

1. Stripe real
2. staging dry run
3. login/session
4. checkout
5. wallet

Entonces el plan cambia a:

1. no publicar el sabado;
2. cerrar blockers;
3. mover release a la siguiente ventana;
4. no compensar metiendo mas horas en features nuevas.

## Recomendacion final

La forma correcta de llegar al sabado es:

1. congelar scope hoy;
2. usar miercoles y jueves para readiness;
3. usar viernes para QA + fixes;
4. usar sabado solo para build y submission.

Si intentamos desarrollar ideas nuevas en paralelo, reducimos mucho la probabilidad real de publicar `1.0` esta semana.
